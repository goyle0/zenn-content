## 18. 新サービス & 最近追加された機能

### AWS App Runner

**概要**: コンテナ化されたWebアプリを最も簡単にデプロイ。ソースコードまたはコンテナイメージから自動デプロイ。

> 🎯 **試験のポイント**: 「コンテナを最小限の設定でデプロイ」「インフラ管理ゼロ」→ **App Runner**

### AWS Proton

**概要**: 開発者向けのインフラストラクチャテンプレート管理。プラットフォームチームがテンプレートを提供。

> 🎯 **試験のポイント**: 「開発者が承認済みインフラを簡単にプロビジョニング」→ **Proton**

### AWS Elastic Disaster Recovery (DRS)

**概要**: オンプレミスやクラウドのワークロードをAWSに継続的にレプリケートしてDRを実現。

> 🎯 **試験のポイント**: 「低コストでDR環境を維持」「RTO分単位」→ **Elastic Disaster Recovery**

### Amazon Location Service

**概要**: 地図、位置追跡、ジオコーディング等の位置情報サービス。

> 🎯 **試験のポイント**: 「位置情報」「地図」「トラッキング」→ **Location Service**

### Amazon Pinpoint

**概要**: マーケティングコミュニケーションサービス。メール、SMS、プッシュ通知のキャンペーン管理。

> 🎯 **試験のポイント**: 「マーケティングメール」「キャンペーン管理」→ **Pinpoint**

### AWS Amplify

**概要**: フロントエンドWebおよびモバイルアプリの構築・デプロイを簡素化。

> 🎯 **試験のポイント**: 「フロントエンド開発者向け」「Web/モバイルアプリ」→ **Amplify**

### Amazon SageMaker

**概要**: 機械学習モデルの構築、トレーニング、デプロイを行うフルマネージドサービス。

| 機能 | 説明 |
|------|------|
| **Studio** | 統合開発環境 |
| **Ground Truth** | データラベリング |
| **Autopilot** | 自動ML |
| **Inference** | モデルのデプロイ |

> 🎯 **試験のポイント**: 「機械学習モデルの構築・デプロイ」→ **SageMaker**

## 19. シナリオ別 解法パターン（詳細版）

### パターン1: 「コスト削減」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| 24時間稼働のEC2 | RI / Savings Plans |
| 中断可能なバッチ処理 | スポットインスタンス |
| 開発/テスト環境 | スポット or オンデマンド（小さめ） |
| アクセスパターン不明のS3 | Intelligent-Tiering |
| ほとんどアクセスしないデータ | Glacier Deep Archive |
| 使われていないEBS/EIP | 削除（Trusted Advisorで検出） |
| 過剰スペックのインスタンス | Compute Optimizer でライトサイジング |

### パターン2: 「高可用性」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| DB障害時の自動フェイルオーバー | RDS Multi-AZ / Aurora |
| アプリの自動復旧 | Auto Scaling + ELB |
| リージョン障害への対策 | クロスリージョンレプリケーション |
| Route 53でフェイルオーバー | ヘルスチェック + フェイルオーバーポリシー |
| S3データの高耐久性 | デフォルトで99.999999999% |
| 静的コンテンツの可用性 | S3 + CloudFront |

### パターン3: 「セキュリティ」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| 最小権限の原則 | IAMポリシーで必要な権限のみ |
| EC2にアクセスキーを埋め込まない | IAMロールを使用 |
| 保存データの暗号化 | SSE-S3 / SSE-KMS / SSE-C |
| 転送中データの暗号化 | HTTPS / TLS |
| SSH不要でEC2に接続 | Session Manager |
| S3バケットのパブリックアクセス防止 | ブロックパブリックアクセス |
| SQLインジェクション対策 | WAF |
| DDoS対策 | Shield（Standard/Advanced） |
| 不正アクセスの検出 | GuardDuty |
| 機密データの検出 | Macie |

### パターン4: 「パフォーマンス」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| 静的コンテンツの高速配信 | CloudFront |
| DBの読み取り負荷分散 | リードレプリカ |
| セッション情報の高速アクセス | ElastiCache |
| DynamoDBの読み取り高速化 | DAX |
| グローバルユーザーへの低遅延 | CloudFront / Global Accelerator |
| S3アップロードの高速化 | Transfer Acceleration / マルチパート |

### パターン5: 「疎結合」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| コンポーネント間の直接呼び出しを避ける | SQS |
| 1つのイベントで複数処理 | SNS + SQS ファンアウト |
| 処理の順序保証 | SQS FIFO / SNS FIFO |
| 複雑なワークフロー | Step Functions |
| イベント駆動アーキテクチャ | EventBridge |

### パターン6: 「移行」系

| シナリオ | 正解の方向性 |
|----------|-------------|
| オンプレサーバーのリフト＆シフト | MGN (Application Migration Service) |
| DBの移行 | DMS |
| 異種DB間のスキーマ変換 | SCT + DMS |
| 大量データのオフライン転送 | Snowball Edge |
| VMware環境の移行 | VMware Cloud on AWS |
| ActiveMQ/RabbitMQの移行 | Amazon MQ |
| Kafkaの移行 | Amazon MSK |

## 20. トラブルシューティング フローチャート

### 🔴 EC2に接続できない

```
EC2に接続できない
    │
    ├─ SSH/RDP → セキュリティグループでポート開放？
    │              └─ はい → NACL で拒否されていない？
    │                          └─ はい → インスタンスの状態は running？
    │                                      └─ はい → パブリックIP または EIP があるか？
    │
    └─ Session Manager → IAMロールに SSMFullAccess が付与？
                          └─ はい → SSM Agent がインストール済み？
                                      └─ はい → VPCエンドポイント または NAT が必要
```

### 🔴 S3にアクセスできない

```
S3にアクセスできない
    │
    ├─ EC2から → IAMロールに S3 権限があるか？
    │              └─ はい → バケットポリシーで拒否されていない？
    │                          └─ はい → VPCエンドポイント経由ならエンドポイントポリシーを確認
    │
    └─ CloudFrontから → OAC/OAI が設定されているか？
                          └─ はい → バケットポリシーで CloudFront を許可しているか？
```

### 🔴 Lambda が動かない

```
Lambda が動かない
    │
    ├─ タイムアウト → 実行時間 > 設定値？
    │                  └─ はい → 設定を延長（最大15分）
    │
    ├─ 権限エラー → 実行ロールに権限があるか？
    │                └─ DynamoDB/S3 などへのアクセス許可を確認
    │
    └─ トリガーされない → リソースベースポリシーで許可されているか？
                          └─ S3イベント/API Gateway からの呼び出し許可を確認
```

### 🔴 RDS に接続できない

```
RDS に接続できない
    │
    ├─ EC2から → セキュリティグループでDB ポート（3306/5432等）を許可？
    │              └─ はい → RDS が同じ VPC またはピアリングで接続？
    │
    └─ Lambda から → Lambda が VPC 内にあるか？
                      └─ はい → RDS Proxy を使用することを検討
                                （接続数の問題を解決）
```

## 22. 試験当日の心得

### ⏱️ 時間配分の目安

| フェーズ | 時間 | 問題数 |
|----------|------|--------|
| 1周目（即答） | 65分 | 65問 |
| 2周目（見直し） | 45分 | マーク問題 |
| 3周目（最終確認） | 20分 | 自信のない問題 |

### 📝 問題を解くコツ

1. **キーワードに注目**
   - 「コスト効率」「最小限の運用」「高可用性」などを見つける

2. **消去法を活用**
   - 明らかに間違いの選択肢を消していく

3. **「全部正しそう」な場合**
   - 最も**シンプル**で**マネージド**な選択肢を選ぶ

4. **迷ったらマーク**
   - 後で見直すためにマークして次へ進む

5. **問題文は最後まで読む**
   - 「最も**コスト効率**の高い」など、条件が最後にあることも

### 🚫 よくある落とし穴

| 落とし穴 | 正解の考え方 |
|----------|-------------|
| 「踏み台サーバーを使う」という選択肢 | Session Manager を検討 |
| 「アクセスキーをコードに埋め込む」 | IAM ロールを使用 |
| 「単一 AZ」という選択肢 | マルチ AZ を選ぶ |
| 「CNAME を Zone Apex に」 | Alias レコードを使用 |
| 「SWF（Simple Workflow）」 | Step Functions（SWFはレガシー） |
| 「Glacier からすぐ取り出せる」 | 取り出しには時間がかかる（Instant Retrieval除く） |
