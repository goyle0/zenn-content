## 1. セキュリティ・ガバナンス

### 1-1. AWS Security Hub

| 項目 | 内容 |
|------|------|
| **概要** | AWS環境全体のセキュリティ状態を一元管理するサービス |
| **主な機能** | セキュリティ検出結果の集約、自動チェック、コンプライアンス評価 |
| **連携サービス** | GuardDuty、Inspector、Macie、IAM Access Analyzer等 |

#### 主要機能

```
┌─────────────────────────────────────────────────────────────┐
│                    AWS Security Hub                         │
├─────────────────────────────────────────────────────────────┤
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐      │
│  │  GuardDuty   │  │  Inspector   │  │    Macie     │      │
│  │  (脅威検知)  │  │ (脆弱性診断) │  │ (機密データ) │      │
│  └──────┬───────┘  └──────┬───────┘  └──────┬───────┘      │
│         │                 │                 │              │
│         └─────────────────┼─────────────────┘              │
│                           ▼                                │
│              ┌─────────────────────────┐                   │
│              │   統合ダッシュボード     │                   │
│              │  ・検出結果の集約       │                   │
│              │  ・優先度付け           │                   │
│              │  ・セキュリティスコア   │                   │
│              └─────────────────────────┘                   │
└─────────────────────────────────────────────────────────────┘
```

#### セキュリティ基準（自動チェック）

| 基準名 | 説明 | 用途 |
|--------|------|------|
| **AWS基礎セキュリティベストプラクティス (FSBP)** | AWSが定めたベストプラクティス | 一般的なセキュリティ評価（推奨） |
| **CIS AWS Foundations Benchmark** | CISが定めた国際標準 | 業界標準への準拠確認 |
| **PCI DSS** | クレジットカード業界のセキュリティ基準 | カード情報を扱うシステム |
| **NIST SP 800-53** | 米国政府のセキュリティ基準 | 政府系システム |

#### 試験のポイント

- **セキュリティの一元管理** → Security Hub
- **複数サービスの検出結果を統合** → Security Hub
- **コンプライアンス状況の可視化** → Security Hub
- **AWS Configが前提条件**（有効化が必要）

### 1-2. AWS Well-Architected Tool

| 項目 | 内容 |
|------|------|
| **概要** | ワークロードをベストプラクティスに照らし合わせてレビューするツール |
| **料金** | **無料** |
| **用途** | アーキテクチャのセルフアセスメント、改善点の特定 |

#### Well-Architected Framework の6つの柱

| 柱 | 英語名 | 主な観点 |
|----|--------|----------|
| 1. **運用上の優秀性** | Operational Excellence | 運用の自動化、継続的改善 |
| 2. **セキュリティ** | Security | データ保護、アクセス制御、インシデント対応 |
| 3. **信頼性** | Reliability | 障害復旧、可用性、自動回復 |
| 4. **パフォーマンス効率** | Performance Efficiency | 適切なリソース選択、スケーリング |
| 5. **コスト最適化** | Cost Optimization | 不要コスト削減、従量課金活用 |
| 6. **持続可能性** | Sustainability | 環境負荷の最小化 |

#### 試験のポイント

- **アーキテクチャの評価・改善** → Well-Architected Tool
- **ベストプラクティスとの比較** → Well-Architected Tool
- **6つの柱の名前と概要を覚える**

## 2. コンテナ関連サービス

### 2-1. Amazon ECR（Elastic Container Registry）

| 項目 | 内容 |
|------|------|
| **概要** | Dockerコンテナイメージを保存・管理するフルマネージドレジストリ |
| **統合** | ECS、EKS、Fargateとシームレスに連携 |
| **セキュリティ** | IAM認証、イメージスキャン、暗号化 |

#### 特徴

```
┌─────────────────────────────────────────────────────────────┐
│                      Amazon ECR                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   開発者 ──push──▶ [コンテナイメージ] ──pull──▶ ECS/EKS    │
│                           │                                 │
│                    ┌──────┴──────┐                         │
│                    │ 機能        │                         │
│                    │ ・イメージスキャン（脆弱性検出）     │
│                    │ ・ライフサイクルポリシー             │
│                    │ ・クロスリージョンレプリケーション   │
│                    │ ・プライベート/パブリックリポジトリ  │
│                    └─────────────┘                         │
└─────────────────────────────────────────────────────────────┘
```

#### パブリック vs プライベート

| 種類 | 用途 | アクセス |
|------|------|----------|
| **ECR Private** | 社内利用のコンテナイメージ | IAM認証が必要 |
| **ECR Public** | OSSなど公開用イメージ | 認証不要でpull可能 |

#### 試験のポイント

- **コンテナイメージの保管** → ECR
- **ECS/EKSと統合したイメージ管理** → ECR
- **脆弱性スキャン** → ECRのイメージスキャン機能

## 3. ネットワーク

### 3-1. AWS Client VPN

| 項目 | 内容 |
|------|------|
| **概要** | リモートユーザーがAWSやオンプレミスに安全にアクセスするためのVPNサービス |
| **プロトコル** | OpenVPN |
| **認証方式** | Active Directory、SAML、証明書 |

#### Site-to-Site VPN との違い

| 項目 | Client VPN | Site-to-Site VPN |
|------|------------|------------------|
| **接続元** | 個人のPC・デバイス | オンプレミスネットワーク全体 |
| **用途** | リモートワーク | 拠点間接続 |
| **スケール** | ユーザー単位 | ネットワーク単位 |

```
【Client VPN】
  リモートユーザー ──VPN──▶ Client VPN Endpoint ──▶ VPC

【Site-to-Site VPN】
  オンプレミスDC ──VPN──▶ Virtual Private Gateway ──▶ VPC
```

#### 試験のポイント

- **リモートワーカーのAWSアクセス** → Client VPN
- **拠点間接続** → Site-to-Site VPN
- **個人デバイスからの安全なアクセス** → Client VPN

### 4-2. Amazon Managed Service for Prometheus

| 項目 | 内容 |
|------|------|
| **概要** | Prometheusのフルマネージドサービス |
| **用途** | コンテナ環境のメトリクス収集・監視 |

```
┌──────────┐    収集    ┌───────────────────┐    可視化   ┌─────────────┐

└──────────┘           └───────────────────┘            │ ボード      │
                                                        └─────────────┘
```

#### 試験のポイント

- **Kubernetesのモニタリング** → Managed Prometheus + Grafana
- **OSSツールのマネージド版が必要** → Managed Grafana/Prometheus

## 5. データ処理・分析

### 5-1. AWS Data Pipeline

| 項目 | 内容 |
|------|------|
| **概要** | データの移動・変換を自動化するETLサービス |
| **用途** | 定期的なデータ処理ワークフロー |
| **状態** | レガシーサービス（新規はStep Functions + Glue推奨） |

#### 代替サービスとの比較

| サービス | 用途 | 特徴 |
|----------|------|------|
| **Data Pipeline** | ETLワークフロー | レガシー、シンプルな処理向け |
| **AWS Glue** | ETL処理 | サーバーレス、現在の主流 |
| **Step Functions** | ワークフロー管理 | 複雑な処理の制御 |

#### 試験のポイント

- 既存システムで使われている可能性があるため出題される
- **新規構築ではGlue + Step Functionsが推奨**

### 5-2. Amazon Elastic Transcoder

| 項目 | 内容 |
|------|------|
| **概要** | 動画ファイルを様々なフォーマットに変換するサービス |
| **入力** | S3に保存された動画ファイル |
| **出力** | 変換後の動画をS3に保存 |

#### MediaConvert との違い

| 項目 | Elastic Transcoder | MediaConvert |
|------|-------------------|--------------|
| **位置づけ** | レガシー | 現在の主流 |
| **機能** | 基本的な変換 | 高度な変換、放送品質 |
| **料金** | 変換時間課金 | 変換時間課金 |

#### 試験のポイント

- **シンプルな動画変換** → Elastic Transcoder
- **放送品質、高度な変換** → MediaConvert

## 6. 最新トピック（2024-2025年の出題傾向）

### 6-1. Amazon Bedrock

| 項目 | 内容 |
|------|------|
| **概要** | 生成AIの基盤モデルをAPIで利用できるサービス |
| **モデル** | Claude、Titan、Llama等の基盤モデル |
| **用途** | チャットボット、文章生成、コード生成等 |

#### 特徴

- **フルマネージド**：インフラ管理不要
- **複数モデル**：用途に応じてモデルを選択可能
- **プライバシー**：データはモデルの学習に使用されない

#### 試験のポイント

- **生成AI機能の追加** → Bedrock
- **基盤モデルのAPI利用** → Bedrock

---

### 6-2. AWS Graviton プロセッサ

| 項目 | 内容 |
|------|------|
| **概要** | AWSが独自開発したARMベースのプロセッサ |
| **メリット** | コスト削減（最大40%）、電力効率向上 |
| **対応インスタンス** | C7g、M7g、R7g、T4g等（末尾がg） |

#### x86 vs Graviton

| 項目 | x86（Intel/AMD） | Graviton（ARM） |
|------|------------------|-----------------|
| **互換性** | 高い（既存アプリそのまま） | 要確認（ARM対応が必要） |
| **コスト** | 基準 | 最大40%削減 |
| **用途** | レガシーアプリ | 新規開発、コスト最適化 |

#### 試験のポイント

- **コスト最適化** → Gravitonインスタンスの検討
- **新規ワークロードで価格性能比を重視** → Graviton

---

### 6-3. Aurora Serverless v2

| 項目 | 内容 |
|------|------|
| **概要** | Aurora のサーバーレス版（自動スケーリング） |
| **スケーリング** | 0.5 ACU〜128 ACU（瞬時にスケール） |
| **v1との違い** | 瞬時のスケーリング、プロビジョンドとの混在可能 |

#### v1 vs v2

| 項目 | Aurora Serverless v1 | Aurora Serverless v2 |
|------|---------------------|---------------------|
| **スケーリング速度** | 数分 | 瞬時（ミリ秒〜秒単位） |
| **最小キャパシティ** | 1 ACU | 0.5 ACU |
| **プロビジョンドとの混在** | 不可 | 可能 |
| **グローバルDB対応** | 不可 | 可能 |

#### 試験のポイント

- **変動の大きいワークロード** → Aurora Serverless v2
- **瞬時のスケーリングが必要** → Aurora Serverless v2
- **最小コストで運用** → Aurora Serverless v2（0.5 ACUから）
